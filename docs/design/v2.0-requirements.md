## 1. Goals & Principles

### 1.1 Primary Goals

| Goal | Description |
|------|-------------|
| **Task Completion** | Ensure all tasks are completed efficiently by available agents |
| **Agent Utilization** | Keep agents productive; avoid idle time when work is available |
| **Reliability** | Detect and recover from agent crashes gracefully |
| **Transparency** | All participants (manager, agents, humans) know the system state |
| **Isolation** | Prevent cross-talk between different agency projects |

### 1.2 Core Principles

1. **One Active Task Per Agent**: An agent works on exactly one task at a time
2. **No Work Interruption**: Never interrupt an agent actively working on a task
3. **Explicit Assignment**: Tasks must be explicitly assigned before work begins
4. **Crash Recovery**: Detect crashed agents and make their tasks available for reassignment
5. **Audit Everything**: Log all significant actions for debugging and accountability

## 2. Core Requirements

### 2.1 Project Structure

```
tmux session: agency-<project-name>     # One session per project
tmux socket: agency-<project-name>      # Isolated socket per project
agency dir: <project>/.agency/         # Configuration and state
```

### 2.2 Entity Cardinality

| Entity | Count | Notes |
|--------|-------|-------|
| Project | 1 | One tmux session per project |
| Manager | 1 | Window index 0, prefixed `[MGR]` |
| Agent | N | Window index 1+, named |
| Task | N | Managed in `var/tasks.json` |

### 2.3 Configuration Requirements

```yaml
# config.yaml (required)
project: <name>                    # Used for session naming
audit_enabled: true                # Enable/disable audit logging
parallel_limit: 2                  # Max concurrent tasks

# manager.yaml (required)
name: coordinator                  # Manager window name
poll_interval: 30                 # Heartbeat poll seconds
auto_approve: false               # Never auto-approve

# agents.yaml (required)
agents:
  - name: coder                   # Must match window name
    config: agents/coder.yaml
```

## 3. Task Lifecycle Rules

### 3.1 Valid Status Transitions

```
                    ┌───────────────────────────────────────┐
                    │                                       │
                    ▼                                       │
┌──────────┐     ┌──────────────┐     ┌──────────────────┐  │
│ pending  │────►│ in_progress │────►│ pending_approval  │──┼──┘
└──────────┘     └──────────────┘     └────────┬─────────┘
    ▲                                          │
    │         ┌──────────┐         ┌───────────┴───────┐
    │         │  failed  │◄────────│   completed       │
    └─────────┤          │         └───────────────────┘
              └──────────┘
```

### 3.2 Status Definitions

| Status | Meaning | Who Sets It |
|--------|---------|-------------|
| `pending` | Task exists, not assigned or not started | TaskStore.add_task(), clear_agent_info() |
| `in_progress` | Agent actively working | TaskStore.pickup_task(), TaskStore.update_task() |
| `pending_approval` | Work completed, awaiting review | TaskStore.complete_task() |
| `completed` | Approved by manager/reviewer | TaskStore.approve_task() |
| `failed` | Rejected; may be retried | TaskStore.reject_task() |

### 3.3 Transition Rules

| From | To | Valid? | Condition |
|------|----|--------|-----------|
| pending | in_progress | ✅ | Task assigned, agent picks up |
| pending | pending_approval | ❌ | Must go through in_progress |
| in_progress | pending | ✅ | Agent abandons (crash recovery) |
| in_progress | pending_approval | ✅ | Agent completes work |
| pending_approval | completed | ✅ | Manager/reviewer approves |
| pending_approval | pending | ✅ | Manager/reviewer rejects (→ failed) |
| completed | pending | ✅ | Manager reopens task |
| failed | pending | ✅ | Agent retries (same as pending) |

### 3.4 Required Task Fields

```json
{
  "task_id": "swift-bear-a3f2",      // Required: unique ID
  "subject": "Fix auth bug",           // Required: brief title
  "description": "...",               // Required: detailed description
  "status": "pending",                // Required: current status
  "priority": "normal",               // Optional: low|normal|high (default: low)
  "assigned_to": "coder",             // Optional: agent name
  "created_at": "2024-01-15T10:00Z", // Auto: ISO timestamp
  "started_at": null,                 // Auto: when in_progress
  "completed_at": null,               // Auto: when completed/failed
  "agent_info": null,                // Auto: {session_id, pid, started_at}
  "depends_on": [],                  // Optional: dependency task IDs
  "result": null,                    // Set: when complete_task() called
  "rejection_reason": null,          // Set: when reject_task() called
  "reviewer_assigned": null          // Set: for pending_approval tasks
}
```

## 4. Heartbeat Processing

### 4.1 Heartbeat Types

| Role | File | Poll Interval | Purpose |
|------|------|--------------|---------|
| Manager | heartbeat.py (manager_heartbeat) | 30s (configurable) | Monitor tasks, spawn agents, detect orphans |
| Agent | heartbeat.py (agent_heartbeat) | 30s (configurable) | Notify agents of assigned tasks |

### 4.2 Manager Heartbeat Loop

```
ON EACH CYCLE (every poll_interval seconds):
│
├─► 1. STALE TASK CHECK
│    ├─ For each task with status="in_progress":
│    │  └─ Check if agent_info.pid exists in system
│    │     ├─ If NOT exists → clear_agent_info(), revert to pending
│    │     └─ Log crash_detected event to audit
│    └─ Continue
│
├─► 2. ORPHAN DETECTION (pending_approval tasks)
│    ├─ For each task with status="pending_approval":
│    │  └─ If reviewer_assigned is null OR reviewer process dead:
│    │     └─ Mark as orphan, clear reviewer_assigned
│    └─ Orphan tasks will get new reviewer on next cycle
│
├─► 3. REVIEW SPAWN
│    ├─ For each pending_approval task without reviewer:
│    │  └─ start_reviewer(agency_dir, task_id)
│    └─ Spawns reviewer agent if configured
│
├─► 4. ASSIGNMENT
│    ├─ Get available agents (capacity < parallel_limit)
│    ├─ Get unblocked pending tasks (dependencies satisfied)
│    └─ assign_tasks_to_agents() → assigns to free agents
│
├─► 5. LAZY SPAWNING
│    ├─ Get tasks assigned but agent not running
│    └─ For each: ensure_agent_running_for_task()
│    └─ Only starts agent when work is assigned
│
├─► 6. SLOT SIGNALING
│    ├─ Track when tasks complete (pending → pending_approval)
│    └─ Release slot via Orchestrator.signal_task_completed()
│
└─► 7. NOTIFICATION (only when conditions met)
     ├─ If unassigned tasks AND in_progress == 0:
     │  └─ Notify manager (once per new task, min 30s apart)
     └─ If pending_approval tasks exist:
        └─ Send approval commands via tmux
```

### 4.3 Agent Heartbeat Loop

```
ON EACH CYCLE (every poll_interval seconds):
│
├─► 1. GET MY TASKS
│    ├─ Fetch tasks assigned to this agent
│    └─ Separate: in_progress vs pending
│
├─► 2. CRASH CHECK
│    └─ Same as manager: check agent_info.pid exists
│
├─► 3. IDLE PING (when pane is idle, every ping_interval)
│    ├─ If has in_progress task AND pane idle AND ping_interval elapsed:
│    │  └─ Send ping: "Still working on {task_id}?"
│    └─ If NO tasks AND pane idle AND 2*ping_interval elapsed:
│       └─ Send ping: "Check agency tasks list"
│
└─► 4. NEW TASK NOTIFICATION
     ├─ If has pending tasks AND NO in_progress AND pane idle:
     │  └─ Notify about highest-priority pending task
     │     "Task ready: {task_id} - {description}..."
     │  └─ Log heartbeat event to audit
     └─ NEVER interrupt an agent with in_progress task
```

### 4.4 Heartbeat Critical Rules

| Rule | Rationale |
|------|-----------|
| **Never interrupt in_progress** | Agent must complete work uninterrupted |
| **Idle detection before ping** | Only ping truly idle agents |
| **Deduplicate notifications** | Track last_notified_task_id to avoid spam |
| **Rate limit pings** | min 30s (manager) or ping_interval (agent) between pings |
| **Crash detection is mandatory** | Prevents orphaned tasks |

## 5. Notification Rules

### 5.1 Notification Triggers

| Trigger | Recipient | Content | Rate Limit |
|---------|-----------|---------|------------|
| Unassigned tasks exist | Manager | `{N} unassigned task(s), {X} agent(s) available` | 30s minimum, only when in_progress=0 |
| Task assigned to agent | Agent (via heartbeat) | `Task ready: {id} - {description}...` | 1 per pending task |
| Agent idle with task | Agent (via heartbeat) | `Still working on {id}?` | Every ping_interval (default 120s) |
| No tasks, agent idle | Agent (via heartbeat) | `Check agency tasks list` | 2*ping_interval minimum |
| Pending approval exists | Manager (via tmux) | `agency tasks approve {id}` | Immediate |
| Task rejected | Original agent | `TASK REJECTED: {reason}` | Via pi-inject or tmux |

### 5.2 Notification Delivery

```python
# Notification flow:
1. Write to var/notifications.json    # For debugging/history
2. Write to var/system_hint.txt       # For load by pi
3. Send via pi-inject socket          # If available and local to agency
4. Fallback: tmux send-keys           # Direct to window
```

### 5.3 Cross-Project Isolation (CRITICAL)

```
┌─────────────────────────────────────────────────────────────────┐
│ SECURITY RULE: Socket Path Validation                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  PI_INJECTOR_SOCKET must be validated:                         │
│                                                                 │
│  IF socket_path NOT within agency_dir:                          │
│     → SKIP injection (prevent cross-project messages)           │
│                                                                 │
│  This prevents agency-project-A from accidentally sending       │
│  notifications to agency-project-B agents.                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.4 Notification Conditions Matrix

| Manager State | Agent State | Manager Notified? | Agent Notified? |
|---------------|-------------|-------------------|-----------------|
| No unassigned tasks | Working | ❌ | ❌ |
| No unassigned tasks | Idle | ❌ | Ping after 2*ping_interval |
| Unassigned + no in_progress | Any | ✅ (once) | ❌ |
| Unassigned + has in_progress | Any | ❌ | ❌ |

## 6. Agent Behavior

### 6.1 Agent States

```
┌─────────────────────────────────────────────────────────────────┐
│                        AGENT LIFECYCLE                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  IDLE ──────────────────────────────────────────────────────►  │
│    │                                                          │
│    │ tasks assigned                                           │
│    ▼                                                          │
│  ASSIGNED ─────────────────────────────────────────────────►  │
│    │                                                          │
│    │ heartbeat notifies, agent picks up                       │
│    ▼                                                          │
│  WORKING ─────────────────────────────────────────────────►   │
│    │                                                          │
│    │ work complete, agency tasks complete                     │
│    ▼                                                          │
│  REVIEWING ───────────────────────────────────────────────►  │
│    │                                                          │
│    │ approved OR rejected                                    │
│    ▼                                                          │
│  IDLE (check for next task)                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 Agent Responsibilities

| Responsibility | Command | Notes |
|---------------|---------|-------|
| Check work queue | `agency tasks my-work` | On start and when idle |
| Pick up task | `agency tasks update {id} --status in_progress` | Sets started_at, agent_info |
| Do the work | (user defined) | Create files, run commands, etc. |
| Complete task | `agency tasks complete {id} --result "..."` | Sets status=pending_approval |
| Handle rejection | Check for `.rejected` file | Read reason, fix issues |

### 6.3 Agent Must NOT

- Start working on a task without marking it `in_progress`
- Complete a task without completing the actual work
- Claim multiple tasks as `in_progress`
- Ignore notification pings about assigned tasks
- Leave tasks in `in_progress` indefinitely

## 7. Manager Behavior

### 7.1 Manager Responsibilities

| Responsibility | How | Frequency |
|---------------|-----|----------|
| Create tasks | `agency tasks add -d "..."` | As needed |
| Assign tasks | `agency tasks assign {id} {agent}` | When agents free |
| Detect crashes | Heartbeat checks PIDs | Every poll_interval |
| Recover orphans | Heartbeat clears reviewer | Every poll_interval |
| Approve work | `agency tasks approve {id}` | When pending_approval |
| Reject work | `agency tasks reject {id} --reason "..."` | When review fails |

### 7.2 Manager Decision Rules

```python
# Assignment Decision
def should_assign_task(task, agent):
    return (
        task.status == "pending" and
        task.assigned_to is None and  # Unassigned
        not is_task_blocked(task) and  # Dependencies complete
        agent.is_free() and            # No in_progress tasks
        not at_parallel_limit()        # Under parallel_limit
    )

# Approval Decision
def should_approve(task):
    return (
        task.status == "pending_approval" and
        work_matches_requirements(task)  # Manager/reviewer judgment
    )

# Rejection Decision
def should_reject(task, reason):
    return (
        task.status == "pending_approval" and
        work_incomplete(task) or
        work_incorrect(task) or
        missing_requirements(task)
    )
```

### 7.3 Manager Must NOT

- Auto-approve tasks (unless `auto_approve: true` in config, and even then...)
- Assign tasks to busy agents (agents with in_progress)
- Assign blocked tasks (dependencies incomplete)
- Start agents without assigned work (lazy spawning)

## 8. Dos and Don'ts

### 8.1 Task Management

| ✅ DO | ❌ DON'T |
|-------|----------|
| Create tasks with clear descriptions | Leave tasks with vague "fix stuff" descriptions |
| Set dependencies explicitly | Assume task order without dependencies |
| Assign tasks explicitly | Let agents pick arbitrary tasks |
| Complete tasks with meaningful results | Mark complete with empty result |
| Reject with specific feedback | Reject with just "bad" |

### 8.2 Heartbeat

| ✅ DO | ❌ DON'T |
|-------|----------|
| Check process existence on every cycle | Skip crash detection cycles |
| Log crash_detected events | Silently ignore crashes |
| Respect idle detection | Ping agents mid-work |
| Rate limit notifications | Spam with every cycle |
| Track last_notification_time | Notify on every unassigned task change |

### 8.3 Notifications

| ✅ DO | ❌ DON'T |
|-------|----------|
| Validate socket paths belong to agency | Send to sockets outside agency dir |
| Write notifications to var/notifications.json | Skip logging |
| Notify manager only when in_progress=0 | Notify when agents busy |
| Notify agents about their assigned tasks | Notify agents about others' tasks |
| Use tmux fallback if pi-inject unavailable | Fail silently |

### 8.4 Review Flow

| ✅ DO | ❌ DON'T |
|-------|----------|
| Set reviewer_assigned on pending_approval | Leave reviewer untracked |
| Check for orphaned reviewers | Assume reviewer always alive |
| Restart agent on rejection | Leave task in limbo |
| Copy rejection reason to task | Lose rejection context |

## 9. Edge Cases

### 9.1 Crash Scenarios

| Scenario | Detection | Recovery |
|----------|----------|----------|
| Agent process dies | PID not found | clear_agent_info(), revert to pending |
| Agent tmux window killed | Window gone, PID dead | Same as above |
| Manager process dies | `.halted` file created | Resume from .halted |
| Reviewer dies | Reviewer PID not found | Clear reviewer_assigned, respawn |

### 9.2 Race Conditions

| Scenario | Handling |
|----------|----------|
| Two agents claim same slot | FileLock on tasks.json |
| Task assigned while agent picking up | Status check before update |
| Approval while agent still working | Only allowed after complete_task() |
| Heartbeat and CLI both modifying | FileLock with 10s timeout |

### 9.3 Dependency Scenarios

| Scenario | Handling |
|----------|----------|
| Task depends on deleted task | Treat as blocking (error) |
| Circular dependencies | Detect in set_dependencies(), reject |
| Dependency task never completes | Task stays blocked forever (human must resolve) |
| Many tasks blocked by one | Human should reassign or complete dependency |

### 9.4 Parallel Limit Scenarios

| Scenario | Behavior |
|----------|----------|
| More agents than limit | Extra agents idle, no work for them |
| All agents busy | New assignments wait for completion |
| Task completes under limit | Slot released, next task assigned |
| Limit changed mid-session | Applies immediately on next heartbeat |

## Appendix D: Security Considerations

1. **Socket Isolation**: `PI_INJECTOR_SOCKET` must be validated against `AGENCY_DIR`
2. **No Secrets**: Never store API keys or passwords in `.agency/`
3. **File Permissions**: `tasks.json` uses FileLock to prevent corruption
4. **PID Validation**: Always verify processes exist before trusting PIDs
